# 意思決定ログ（Q&A 形式）

本ドキュメントは、プロジェクト開始から初回デプロイまでの意思決定の経緯を時系列で記録したものである。

---

## 1. アプリの目的・ユースケース

**Q: どんなアプリを作るのか？**

A: LINE 公式アカウントのリッチメニューから起動する LIFF アプリ。チャット風 UI のステップフォームで、Bot が定型の質問を順番に投げかけ、ユーザーが選択肢から回答する。サンプル／プロトタイプとして開発し、まずは Vercel 上で動作確認できる状態をゴールとする。

**Q: 会話相手は AI チャットボットか？**

A: いいえ。リアルタイムの会話ではなく、定型的な質問を投げかけてユーザーが選択肢から回答する「フォーム送信」に近い。途中でアプリを終了しても、再開時に途中まで入力されたものが残る必要がある。

**Q: 質問の内容テーマは？**

A: 特定テーマに縛らず、様々な質問型をデモする汎用デモとした。

---

## 2. 技術選定

**Q: なぜ最初 Next.js を検討し、React（Vite）に変更したのか？**

A: 当初は Next.js（フルスタック）を選定したが、以下の理由でオーバースペックと判断し React + Vite に変更した。

- SPA で完結する（画面 1 つ、ルーティング不要）
- LIFF アプリは LINE 内 WebView で動くため SSR/SSG の恩恵（SEO 等）がない
- 認証後にしかコンテンツを表示しないため SSR しても意味がない
- App Router の機能（Server Components, Middleware, ISR）をほぼ使わない
- Vite の方がビルドが速く、バンドルサイズも小さい（LIFF の表示速度に直結）
- 本番 AWS 移行時にフロント・バックエンド分離がしやすい

ユーザーから「将来サーバーからデータ取得する」との指摘があったが、クライアントから API を fetch するだけなので React 単体で問題ないと結論づけた。

**Q: Webhook はどこに置くのか？**

A: Vercel Serverless Functions として `src/api/webhook.ts` に配置。Vercel のルートディレクトリを `src` に設定する前提で、Vercel から見ると `api/webhook.ts` として認識される。フロントエンドの Vite ビルドからは `tsconfig.json` の `exclude` で除外。

**Q: データ保存はどうするのか？**

A: サンプルでは localStorage。本番ではバックエンド API と繋いで DB に保存する予定だが、今はプロトタイプを LINE 上から見れる状態にすることが優先。

## 3. LINE プラットフォームの設定

**Q: LIFF アプリはどこに作成するのか？**

A: 当初 Messaging API チャネルに LIFF を追加しようとしたが、2019年の仕様変更により **LIFF は LINE ログインチャネルにしか追加できない**。そのため、同一プロバイダー内に以下の 2 チャネルを作成した。

- **Messaging API チャネル**（Sample ChatBot）: Webhook、Channel Secret、Channel Access Token
- **LINE ログインチャネル**（LIFF Sample）: LIFF アプリ（Chat Form）を登録、LIFF ID を取得

**Q: `liff.sendMessages()` でお礼メッセージを送る予定だったが、なぜ変更したのか？**

A: `liff.sendMessages()` は「ユーザーが自分のトークに送信する」動作であり、Bot からのメッセージにならない。Bot からユーザーにメッセージを送るには Messaging API の `pushMessage` を使う必要があるため、サーバーサイド（`/api/complete`）経由で Push API を呼ぶ方式に変更した。

**Q: `liff.sendMessages()` 使用時に `The permission is not in LIFF app scope` エラーが出たのはなぜ？**

A: LIFF アプリの Scope 設定で `chat_message.write` が有効になっていなかったため。さらに、LINE ログインチャネルが「開発中」のままだと権限が制限されるため、チャネルを「公開」に変更する必要があった。

**Q: Bot から定型の応答メッセージ（「このアカウントでは個別のお問い合わせを～」）が返ってきたのはなぜ？**

A: LINE Official Account Manager の「応答メッセージ」がデフォルトで ON になっていたため。OFF にすることで解消。

## 4. デザイン

**Q: デザインの方針は？**

A: パステルピンク（#FFB5C5）基調のポップ調。シンプルで親しみやすく、スマホ利用を前提としたユーザー体験の高い高品質なもの。

**Q: LIFF の表示モードは？**

A: Full（全画面表示）。リッチメニューから起動。

**Q: フォントは？**

A: 「M PLUS Rounded 1c」を採用。丸みのある書体がポップ調のデザインに合致するため。

---

## 5. 質問フロー

**Q: 質問フローに条件分岐は必要か？**

A: 必要。回答内容に応じて次の質問が分岐する。サンプルでは 2 箇所の分岐を設定（趣味でスポーツを選択した場合 / SNS 利用頻度で毎日を選択した場合）。

**Q: 質問データの管理方法は？**

A: JSON ファイル（`src/data/questions.json`）で管理。コードに直接書かず、JSON の編集のみで質問の追加・変更が可能。

---

## 6. 開発環境

**Q: プロジェクト構成で `src/` の中に `package.json` を置いたのはなぜ？**

A: ユーザーの指示により、React プロジェクトを `src/` ディレクトリ配下に閉じて構築した。リポジトリルートにはドキュメント（CLAUDE.md, README.md, docs/）のみを配置し、フロントエンドのコードとビルド設定を `src/` に集約している。

**Q: Node.js のバージョン問題はどう解決したか？**

A: 当初 Vite 8 をインストールしたが、Node.js v20.11.0 では動作しなかった（v20.19.0+ が必要）。nvm での Node.js アップデートが Windows 環境の権限問題で困難だったため、Vite 6 にダウングレードして解決した。

**Q: ローカルで LIFF を動作確認するにはどうしたか？**

A: LIFF のエンドポイント URL を `https://localhost:5175` に設定し、Vite に `@vitejs/plugin-basic-ssl` を追加して HTTPS で dev server を起動した。LIFF は HTTPS が必須だが、localhost は例外的に自己署名証明書でも動作する。

---

## 7. デプロイ

**Q: デプロイ方法は？**

A: GitHub リポジトリと Vercel を連携し、Push で自動デプロイ。`npx vercel` の CLI デプロイではなく、Vercel ダッシュボードから GitHub リポジトリをインポートする方式を採用（ユーザーの提案）。

**Q: Vercel の Root Directory 設定は？**

A: `src` を指定。これにより Vercel はビルドコマンドを `src/` 内で実行し、`src/api/` を Serverless Functions として認識する。

**Q: 本番のデプロイ先は？**

A: 最終的には AWS を想定。ただし現段階では Vercel でプロトタイプの動作確認を行う。フロントエンドとバックエンドが分離されている構成のため、AWS 移行は容易。
